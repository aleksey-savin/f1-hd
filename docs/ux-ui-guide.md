## Стек и базовые принципы

- **React + react-router** (data router: `loader`/`action`, `useFetcher`,
  вложенные маршруты) — не меняется миграцией.
- **Целевой UI-стек: Tailwind v4 + shadcn/ui** (см. раздел миграции ниже) —
  им строятся все новые и мигрированные экраны. **react-bootstrap** остаётся
  только на немигрированных экранах; там же остаются его конвенции
  (`Row`/`Col`, утилиты `mb-3`…).
- **react-icons**, набор **Remix (`Ri*`)** — единственный источник иконок
  приложения (`RiSaveLine`, `RiAddFill`, `RiDeleteBinLine`, `RiRefreshLine`,
  `RiCheckLine` …); lucide живёт только внутри сгенерированных `components/ui`.
- **zustand** для UI-состояния (`store/offcanvas.js`, `store/toast-store.js` …),
  **Context** для темы (`store/theme-context.jsx`; он же ставит класс `.dark`
  на `<html>`).
- **react-select** только через обёртку `UI/Select.jsx` (стилизована
  css-переменными новой темы, меню в портале) — голый `react-select` не
  использовать. Обёртка общая для обеих систем до замены на Combobox.
- Отдельные ветки **`BrowserView` / `MobileView`** (`react-device-detect`) там,
  где десктоп и мобайл расходятся по раскладке.
- Весь UI-текст — **на русском**. Семантику передаём вариантами компонентов и
  токенами (`primary`/`success`/`destructive`…), а не произвольными цветами.

## Миграция на Tailwind v4 + shadcn/ui (идёт с 2026-07)

Приложение поэтапно переезжает с react-bootstrap на Tailwind v4 + shadcn/ui
(области мигрируют по одной; layout сохраняется, поверхности — в духе
согласованного макета: панели с тонкими границами на канве, без теней).
Эталоны: **списки** — справочник «Вендоры» (`pages/Vendor/*`, пилот),
**настройки** — «Мой аккаунт» (`pages/User/MyAccount.jsx`).
Пока в коде живут обе системы, разделы гайда про react-bootstrap остаются в
силе для немигрированных областей.

**Инфраструктура:**

- Токены темы — `src/styles/tailwind.css` (`:root`/`.dark`; класс `.dark` на
  `<html>` ставит `ThemeProvider`). Радиус `0.5rem`, шрифт — **Inter**
  (self-hosted `@fontsource-variable/inter`), он же включён и для
  легаси-экранов через `html:root { --bs-font-sans-serif }`.
- Все tailwind-утилиты — с префиксом **`tw:`** (`tw:flex`, `tw:hover:bg-accent`,
  `tw:dark:border-input`) и **только из встроенной сетки** — произвольных
  значений вида `text-[15px]` не пишем. Префикс снимется автозаменой в конце
  миграции. `cn()` — из `@/lib/utils` (twMerge знает префикс).
- shadcn-компоненты — `src/components/ui/*.tsx` (генерирует CLI, префикс
  ставится сам): Button/Badge/Alert расширены вариантами `success`/`warning`/
  `info` (+ `light` у Alert). Сгенерированный код руками не перекраиваем —
  расхождения с доками shadcn чинятся токенами или точечно (см. ниже).
- **Preflight ВЫКЛЮЧЕН до эндшпиля** — из этого следуют пять ловушек, уже
  наступленные и закрытые:
  1) браузерные дефолты `<button>`/`<a>` никто не сбрасывает — интерактивные
     элементы задают `appearance/border/bg/no-underline` (и `p-0` у
     кнопок-контролов фиксированного размера: switch/checkbox/radio) явно;
  2) вендоренный bootstrap уложен в css-слой `layer(bootstrap)` — без этого
     его reboot перебивал бы tw-утилиты;
  3) типографику и отступы в tw-коде задаём явно (`tw:space-y-*`, `tw:my-0`);
  4) голый `tw:border` (так пишут сгенерированные shadcn-компоненты) без
     preflight падал в `currentColor` — светлые рамки меню в тёмной теме;
     восстановлено правилом `*,::before,::after { border-color: var(--border) }`
     в `@layer base` (`styles/tailwind.css`) — только цвет, width/style
     остаются за bootstrap reboot;
  5) `tw:border-dashed`/`tw:border-solid` — это border-STYLE-шорткаты, ставят
     стиль всем 4 сторонам. Без preflight-сброса `border-width:0` стороны без
     явной ширины рисуются дефолтной (medium ≈ 3px): `tw:border-t
     tw:border-dashed` даёт не верхнюю линию, а целый бокс. Односторонний
     пунктир задаём инлайном (`style={{ borderTop: "1px dashed var(--border)" }}`),
     не классами (эталон — `ServicePlan/View.jsx`, разделитель нерабочего
     времени).
- z-index: radix-оверлеи подняты до 1100 (блок `[data-slot=…]` в
  `tailwind.css`), выше bootstrap-модалок (1045–1055). Бар оболочки —
  инлайн `zIndex: 1030` (легаси-шкала: выше рейла статусов 1020, ниже
  модалок) — tw-сетка z заканчивается на 50.
- **Скролл-лок radix**: `ui/dropdown-menu` переведён на `modal={false}` по
  умолчанию (меню не должно лочить скролл). Модальные оверлеи
  (Dialog/AlertDialog/Sheet) лочат законно — «прыжки» лэйаута сняты
  системно: на десктопе желоб скроллбара есть всегда (`html { overflow-y:
  scroll }`), а компенсирующий `padding-right` от react-remove-scroll погашен
  (`body[data-scroll-locked] { padding-right: 0 !important }`) — при локе
  ничего не сдвигается ни на пиксель. Пиксельные компенсации фиксед-элементов
  не использовать — они не покрывают фоновую картинку и полосят.
- **Порталы внутри модальных radix-оверлеев некликабельны** (pointer-events:
  none на body + закрытие по «клику снаружи»). `UI/Select` в `FormSheet` и в
  Sheet фильтра переключается на инлайн-меню через `InsideOverlayContext`
  (`components/app/overlay-context.ts`) — новые модальные обёртки со своими
  порталами обязаны ставить этот Provider.
- **Бар статусов сотрудников** (`components/User/WorkStatusBar.jsx`) — на tw
  по согласованному мокапу: rail 72/336px (свёрнутый — radix-Tooltip влево,
  раскрытый — группы цветом статуса, время `tabular-nums`, футер «Обновлено …»
  от max(updatedAt), не от «сейчас»), мобильная лента с градиентом и панелью
  чипов. **Палитра статусов — per-theme токены `--ws-st-*`**
  (`styles/tailwind.css`): каталог `util/work-statuses.js` отдаёт
  `color: "var(--ws-st-…)"` — годится только для CSS-контекстов; копии
  каталога у бота/бэкенда остаются на hex. В `index.css` от бара осталось
  только вне-сеточное: `.has-ws-rail` (резерв 92px), `.ws-rail-scroll`
  (тонкий скроллбар), `.ws-strip-scroll` (скрытый), `.ws-avatar`/`.ws-live`.
- Мигрированные маршруты перечислены в `layout/Root.jsx` и рендерятся **на
  канве без bootstrap-Card**; из-за fixed-слоя фоновой картинки **любой контент
  канвы обязан быть `position: relative`** (контент-обёртка — уже в Root;
  `Footer` — сам `tw:relative` и при картинке становится плашкой-«листом»),
  иначе fixed-слой рисуется поверх и элемент «пропадает». Если у пользователя задана
  **фоновая картинка**, Root кладёт контент мигрированного маршрута на «лист»
  цвета канвы (`var(--bs-body-bg)`, скругление + тонкая граница + `p-4` —
  паттерн content sheet on wallpaper). Лист **обнимает контент, а не тянется
  на всю ширину**: `maxWidth` листа = контентная ширина страницы (её
  `tw:max-w-*`) + паддинг, значения — в конфиге `MIGRATED_ROUTES` в Root
  (новый маршрут добавляется туда парой «путь + ширина»). Обои остаются
  видимыми по бокам, пустых полей-«карточек» нет. Без картинки лист не
  рисуется — экраны про фон не знают.
- **Глобальный автоскейл `<img>`** (`index.css`: `img { width/height: auto
  !important; … }` — легаси-хак для картинок в описании тикета) перебивает
  tw-размеры на любых `<img>`. Фиксированные превью/аватары в мигрированном
  коде — `div`/`span` с `background-image` + `tw:bg-cover tw:bg-center`
  (эталон — `AccountSettings/BackgroundImageUpload.jsx`). Заскоупить хак —
  задача эндшпиля.

**Каталог компонентов `src/components/app/`** (двойники примитивов приложения;
эти — вместо ручной сборки и вместо легаси `src/UI/*`):

| Компонент | Назначение |
|---|---|

| `FormSheet` | Нижняя шторка форм: десктоп — колонка по центру (по умолчанию 672px, `wide` → 896px для форм со сводкой), мобайл — почти весь экран. **На десктопе клик мимо/Escape НЕ закрывают** (защита от потери данных). Для любых «всплывающих» форм |

`components/app`, а не копируется по страницам.

**Правила:**

- **Каждый экран миграции начинается с HTML-макета** (артефакт: десктоп +
  мобайл, обе темы), который согласовывается с Алексеем до написания кода;
  код повторяет макет пиксельно. Эталон языка — макет «Вендоров»: группы-метки
  со счётчиками, монограммы, цветной статус-текст вместо заливных бейджей,
  бирюзовое главное действие.
- Мигрированный файл: **ни одного импорта react-bootstrap и ни одного
  bootstrap-класса**; утилиты только `tw:*`, компоненты — shadcn/каталог выше.
- В легаси-экраны `tw:`-классы не добавлять; новые фичи в мигрированных
  областях — сразу на tw, в немигрированных — по-старому.
- Radix-оверлеи не открывать из bootstrap-модалок (и наоборот).
- `UI/Select` (react-select) остаётся общим для обеих систем до эндшпиля.
- **Верификация**: `pnpm build` + `pnpm eslint` — и передать Алексею на
  проверку в живом UI (он проверяет сам и говорит, если что-то не так).
  Playwright-проходку самостоятельно не устраивать — только по явной просьбе
  (рецепт: mint JWT в `hd-backend-1`, `addInitScript` с localStorage).

## Когда что использовать (главное правило выбора контейнера)

| Ситуация | Контейнер (мигрированные экраны / легаси) | Референс |
|---|---|---|
| Подтверждение, короткий вопрос, мелкий диалог (1–3 поля) | **AlertDialog/Dialog** / Modal `centered` | `components/app/DeleteItem.tsx` / `components/DeleteItem.jsx` |
| Заполнение/редактирование формы (add/update сущности) | **Нижняя шторка**: `app/FormSheet` (в `app/ListWrapper` встроена) / Offcanvas снизу | `pages/Vendor/*` / `UI/ListWrapper.jsx` |
| Большая многошаговая форма (мастер) | **Расширенная** шторка + степпер во всю ширину + живая сводка справа | `ServicePlan/Form.jsx` (tw) / `ClientDevice/WizardStepper.jsx` |
| Боковые фильтры списка (мобайл) | Sheet сбоку (в `app/ListWrapper`) / Offcanvas сбоку | `components/app/ListWrapper.tsx` |
| Предпросмотр записи списка/дерева («заглянуть», не уходя) | **Шторка справа**: крошки/статус/ключевые поля + вложенная навигация; главное действие — «Открыть карточку» | `components/Location/PreviewSheet.jsx` |
| Список/таблица как «всплывающая» панель | Шторка снизу | `layout/Dashboard/Offcanvas.jsx` (легаси) |
| Правка **документа** (заметка, статья) | **На месте, на той же странице** | `components/KnowledgeBase/NoteView.jsx` |
| Массовые действия над выделением | Плавающая панель (десктоп) + `UI/MobileActionBar` | `components/Ticket/BulkActionBar.jsx` |

Коротко, четыре опорных правила:

1. **Диалог — только для маленьких вещей**: подтверждения, единичный вопрос,
   быстрое действие. Не размещать в диалоге полноценные формы.
2. **Заполнение форм — в шторке, выезжающей снизу** (на десктопе клик по
   свободной области и `Escape` её не закрывают — защита от потери данных).
3. **Большая многошаговая форма — со степпером во всю ширину шторки и живой
   сводкой справа** (расширенная шторка; эталон — `ServicePlan/Form.jsx`).
4. **В тёмной теме не использовать текст и иконки цвета `info`** — низкий контраст,
   плохо читается (см. раздел «Тема и цвет»).

## Раскладка страницы

`layout/Root.jsx` задаёт каркас:

- **Десктоп** (`BrowserView`): фиксированный **бар оболочки на tw**
  (`layout/Navbar.jsx`, Фаза 2): панель поверхности 56px с тонкой нижней
  границей в обеих темах; бренд — лого компании (`contacts.logo` из настроек,
  локальный файл) или текст «Help**Desk**»; пункты и разделы — из конфига
  `layout/Navigation/menu.js` (иконки только `Ri*`; раздел виден, если виден
  хотя бы один его пункт); на **< xl** меню сворачивается в бургер-Sheet
  (`layout/NavDrawer.jsx`, общий с мобильным шеллом). Меню пользователя —
  **Popover, не DropdownMenu** (внутри инпут заметки статуса, radix-меню
  ломает его typeahead'ом). Контент — `paddingTop: 100px`; слева опционально
  колонка-сайдбар (`leftSidebarContent`). `Container fluid`,
  `maxWidth: 1920px`, отступы `px-5`. Переходы — `animations/Transition`.
  **Мигрированные маршруты** (список в Root) рендерятся прямо **на канве без
  bootstrap-Card** — панель рисует сам экран (`app/ListWrapper`); легаси-экраны
  по-прежнему живут в `Card.shadow`.
- **Мобайл** (`MobileView`): нескроллящийся app-shell — см. раздел
  «Мобильная навигация».
- Десктоп и мобайл разведены через `BrowserView`/`MobileView` (`react-device-detect`,
  по User-Agent).
- Тосты — в `ToastContainer position="bottom-end"` (см. «Обратная связь»).

Списочные страницы: **мигрированные** — через `components/app/ListWrapper`
(заголовок + счётчик рядом, справа поиск / сортировка-текст-дропдаун / чипы /
«Добавить …» с текстом; на мобайле «Добавить» — квадратная иконка, поиск всегда
виден); **легаси** — через `UI/ListWrapper.jsx` (заголовок, «Обновить»/«Назад»/
«Добавить», поиск, сортировка-`Select`, «Найдено: N»). Пустое состояние в обоих —
`AlertMessage variant="light"`.

## Мобильная навигация (app-shell)

На мобайле (`MobileView` в `layout/Root.jsx`) весь UI — **нескроллящийся app-shell**
`.mobile-shell` (`position:fixed; height:100dvh; display:flex; flex-direction:column;
overflow:hidden`). Скроллится только средняя область `.mobile-shell__scroll` (`<main>`);
верхний бар и нижний таб-бар — её соседи по флексу, **а не `position:fixed`**. Это
структурно убирает баг iOS Safari, где фиксированные панели «всплывали» посреди экрана
при инерционном скролле. Десктоп (`BrowserView`) сохраняет прежнюю схему (`fixed-top`
`Navbar` + `paddingTop:100px`).

```text
.mobile-shell                         position:fixed, 100dvh, flex-column, overflow:hidden
├─ <NavigationBar embedded />         верхний бар: .navbar.mobile-shell__header (в потоке)
├─ <main .mobile-shell__scroll>       единственный скролл-контейнер
│    └─ <Container> <Outlet/> <Footer/> </Container>
└─ <MobileBottomNavbar />             плавающий «остров»: .mobile-tabbar (absolute)
```

**Нижний таб-бар — `layout/MobileBottomNavbar.jsx`** («плавающий остров»):

- Скруглённый бар с `backdrop-filter: blur`, парит над краем
  (`position:absolute; bottom: calc(10px + env(safe-area-inset-bottom))`).
- Вкладка = иконка (`Ri*`/`Tb*`) + короткая подпись; тач-таргет ≥ 44px.
- Активная вкладка — бирюзовый акцент (`--bs-success`) и пилюля, которая **плавно
  скользит** между вкладками через `framer-motion` `layoutId` + `LayoutGroup`
  (с уважением `useReducedMotion`). Пилюлю центрируем через `inset`, а **не**
  `transform`: framer сам управляет `transform`, иначе выделение «съезжает».
- Набор вкладок — конфиг-массив, фильтруется по правам/`isEndUser` (как в меню).
  Активность считается явно; маршруты взаимоисключающие → ровно одна активная
  вкладка (инвариант: одна пилюля на `layoutId`).
- Обязательны `aria-label` и `aria-current` на вкладках (иконки — `aria-hidden`).

**Верхний бар (мобайл)** — тот же `NavigationBar` с пропом `embedded`: статичный
флекс-ребёнок (`.mobile-shell__header`, safe-area-паддинги в `index.css`) на
tw-классах. Гамбургер + бренд + чип статуса (Popover с `WorkStatusSwitcher`).

**Бургер-меню** — `layout/NavDrawer.jsx` (Sheet слева, общий для мобильного
шелла и узкого десктопа): шапка с аватаром и ролью, скруглённые строки
навигации из конфига `menu.js` (разделы — `Collapsible`), футер —
`app/ThemeSegment` + «Мой аккаунт» + «Выйти». Легаси `.drawer-*`-CSS удалён.

**Правила (важно):**

- **Не вешать слушатели скролла на `window`** на мобайле — он не скроллится (гейт
  `isBrowser` в `UI/BackToTop.jsx`); нужен скролл — слушать `.mobile-shell__scroll`.
- Панели навигации на мобайле — **никогда не `position:fixed`** (только флекс/absolute
  внутри shell), иначе вернётся «дрейф».
- Сброс скролла при навигации — вручную (`mobileScrollRef` + `useEffect` по
  `location.pathname` в `Root.jsx`; router `ScrollRestoration` не используется).
- Тема навигации — scoped-токены `--nav-*` / `--tab-*` / `--tabbar-*` в
  `index.css` теперь **ссылаются на токены новой системы**
  (`var(--border)`/`var(--card)`/`var(--primary)`) — тёмные значения даёт класс
  `.dark`, отдельных переопределений в `dark-theme.css` больше нет.
- `index.html`: `viewport-fit=cover` + `theme-color`; safe-area учитывается и сверху,
  и снизу.

## Страница сущности (карточка): hero → секции → действия

Детальные страницы (устройство, пользователь, компания, заметка базы знаний)
строятся по одному шаблону. Легаси-эталон — **`components/ClientDevice/View.jsx`**;
**tw-эталон (новая система)** — **`components/ServicePlan/View.jsx`** (первая
мигрированная карточка, макет согласован до кода).

**tw-карточка (мигрированные экраны):**

- **Возврат к списку — хлебные крошки сверху** («‹ Услуги», Link на список), а
  НЕ кнопка «К списку» в действиях (дублировала бы). Крошки — по умолчанию для
  всех view-экранов.
- **Hero**: иконка/монограмма-плитка (`size-14 rounded-2xl bg-accent
  inset-ring`, та же, что в строке списка) · `h1` · «статус» строкой (цветной
  текст с точкой) · охват счётчиками (`tabular-nums`) · справа **одна залитая
  «Изменить»** + меню «⋯» (пока только Удалить, через `app/DeleteDialog`,
  рендерится вне radix-меню). Подписи «Действия» нет.
- **Секции** — панели на канве (`rounded-xl border border-border bg-card p-5`)
  с uppercase-eyebrow-меткой (+ счётчик), примитивы `Panel`/`Eyebrow`/`Pill`
  объявляем прямо в файле. Однородные списки (категории, компании) —
  **нейтральными пилюлями** (не раскрашиваем), показываем **полностью**.
- **Правка — в нижней шторке** прямо на карточке: `FormSheet` + `<Outlet/>` +
  `store/offcanvas` (как в `ListWrapper`); маршрут `update` вложен в `:id`.
- **Скролл к началу при монтировании** (`useEffect(() => window.scrollTo(0,0),
  [])`): на десктопе Root сбрасывает только мобильный контейнер, а window-скролл
  при навигации сохраняется — без сброса hero карточки, открытой из
  проскроленного списка, уезжает под фиксированный бар. Глобально сбрасывать по
  `pathname` нельзя — сломает фон под шторками-формами (их путь тоже меняется).
- **Строка списка ведёт на карточку** (не в шторку правки): `app/ListRow` проп
  **`detailTo`** (клик → `navigate(detailTo)`, доступно всем, кто видит список;
  «⋯» с правкой/удалением остаётся под `canManage`). В Root `MIGRATED_ROUTES`
  под общим префиксом заводим ДВЕ записи, если у списка и карточки разная
  ширина: `/…/` (со слэшем, карточка `max-w-4xl` → 944) матчится в `.find`
  раньше `/…` (список `max-w-7xl` → 1328).

**Легаси-карточка (bootstrap):**

```text
.account-hero           медиа-объект: плитка/аватар · заголовок + статус · действия справа
└ вкладки (опц.)        Tabs.scrollable-tabs, активная вкладка в URL (?tab=)
  └ Card-секции         SectionCard: .cap-card-title + строки .contact-row
     └ .contact-row     иконка-плашка · МИКРО-ПОДПИСЬ · значение (mono для серийников/IP)
низ: border-top ряд     кнопки в Col sm="auto" + w-100 (на мобиле — в столбик)
```

- **Два примитива, объявляемые прямо в файле страницы** (не абстрагируем раньше
  времени): `SectionCard` (`Card.border-0.shadow-sm.h-100` + `.cap-card-title`)
  и `Line` (`.contact-row`). Пустое значение — общий `—` (`text-body-secondary`),
  а не пустая строка.
- **Вкладки — только когда у сущности есть независимые «режимы работы»**
  (карточка / мониторинг / конфигурации). Документ (заметка, статья) вкладок не
  требует: одна колонка сверху вниз читается лучше.
- **Пустое состояние внутри секции — не пустота**: крупная приглушённая иконка,
  заголовок, объясняющий абзац (`max-width: 480`) и главное действие. Если прав
  не хватает — вместо кнопки строка «Недостаточно прав — обратитесь к
  администратору».

## Действия на странице сущности

**Ровно одна залитая кнопка**, и её выбирает состояние, а не порядок в разметке.
Пример из базы знаний (`components/KnowledgeBase/NoteActions.jsx`): модератору
непроверенной заметки главное действие — «Проверить», всем остальным —
«Редактировать». Вторая кнопка — `outline-secondary`. Всё редкое и опасное — в
icon-only меню `⋯` (`RiMoreLine`, `Dropdown` с `Dropdown.Header`-группами,
разрушающее последним и `text-danger`).

- **Подпись «Действия» запрещена.** Она ничего не сообщает: пользователь обязан
  открыть меню, чтобы узнать, что там. Icon-only «⋯» честнее и занимает меньше.
- **Чужой запрос, ожидающий решения, — инлайн-`Alert`, а не пункт меню.**
  «Пётр Петров запросил удаление 3 июля» + `Подтвердить` / `Отклонить` рядом с
  контекстом (`components/KnowledgeBase/PendingRequestAlert.jsx`). Решение,
  спрятанное в меню, принимается вслепую: не видно ни кто попросил, ни когда.
  Кнопки внутри цветного алерта — `.alert-action-btn` (контраст по `currentColor`).
- Кнопка действия и её результат называются **одним словом**: «Проверить» →
  «Заметка отмечена как проверенная» → бейдж «Проверено».
- **`overflow: hidden` на блоке, где живёт кнопка меню, обрежет меню.** Обычно
  его ставят, чтобы фон-градиент не вылез за скругление, — но фон и так
  клипается по `border-radius`. Если `overflow` действительно нужен, дропдаун
  должен рендериться в портал.

## Статус, который протухает, — предложение, а не бейдж

Если у состояния есть **автор, дата или срок**, бейджа мало. Пишем фразу:

```text
✓ Проверено · Иванов И. · 16.06.2026 · действует ещё 24 дн.
⚠ Не проверено · изменил Петров П. · 03.07.2026
```

Эталон — `components/KnowledgeBase/VerificationLine.jsx` (`.kb-trust`). Цветом
красим только иконку и само состояние; автор и дата — `--bs-secondary-color`.
Срок на исходе переключает тон на `warning`, даже если состояние «хорошее».
Бейдж (`Badge`) остаётся для компактных списков, где нет места на фразу.

Чтобы фраза была возможна, бэкенд обязан отдавать **имена**, а не `ObjectId`:
`populate` актора в `getOne` и в ответах мутаций (см.
`backend/controllers/knowledgeNote.js`, `ACTOR_PATHS`). В списках имена не нужны
— там достаточно иконки, и populate туда не тащим.

## Правка на месте (Obsidian-режим)

Когда страница — **документ**, а не форма, правим его на месте, а не в Offcanvas.
Эталон — `components/KnowledgeBase/NoteView.jsx`.

**Главное правило: между чтением и правкой не должно двигаться ничего**, кроме
того, чем управляют. Как это достигается:

- Заголовок в правке — тот же `h2`, только `Form.Control` без рамки, с теми же
  метриками (`.kb-title` / `.kb-title-input`).
- Свойства — `.contact-row`-строки, у которых значение имеет `min-height` под
  высоту `UI/Select`: пилюли в чтении и селект в правке занимают один бокс.
- Тело — один и тот же класс контента у вьюера и редактора
  (`.toastui-editor-contents`), поэтому мера чтения задаётся один раз.

**Клавиши и триггеры:** `Ctrl/Cmd+E` — переключить режим, `Ctrl/Cmd+S` —
сохранить, `Esc` — выйти. Двойной клик по тексту открывает правку в том же месте
(гейт по правам, не в архиве; клики по ссылкам и чекбоксам task-list
игнорируются). Точную каретку в WYSIWYG не обещаем: `EditorPos` там — offset
ProseMirror-документа, поэтому целимся в начало блока через DOM `Range`.

**Несохранённые изменения:** `useBlocker` (роутер данных) + `beforeunload` +
`ConfirmActionModal`. Собственные переходы (сохранили новую сущность, отменили
создание) обходят блокировку через ref-флаг.

**Автосохранения нет, если у сохранения есть побочный эффект.** Правка заметки
сбрасывает отметку «Проверено» — о таком побочном эффекте предупреждаем **до**
сохранения, прямо в строке статуса: «Сохранение снимет отметку „Проверено"».

## Контекстный остров действий на мобилке

`UI/MobileActionBar.jsx` — общий компонент (заявки: bulk-действия; база знаний:
bulk-модерация и правка заметки). Портал в `.mobile-shell`, классы
`.mobile-actionbar*` в `index.css`.

- **На экране всегда ровно один плавающий объект.** Пока панель в DOM,
  `.mobile-shell:has(.mobile-actionbar) .mobile-tabbar { visibility: hidden }` —
  таб-бар уступает ей место, «бутерброд» из двух плашек невозможен структурно.
- **Заблокированное действие остаётся нажимаемым и объясняет причину**: на
  тач-экране hover-тултипа нет, поэтому тап на 4 секунды подменяет строку статуса
  причиной (`reason`). Причина называет объекты **по имени** (заголовок заметки,
  номер заявки), а не по внутреннему id.
- Один массив `actions[{ key, icon, label, reason, danger }]` — источник правды и
  для десктопной плавающей панели, и для мобильного острова.
- Цветом выделено ровно одно действие (обычно разрушающее, `is-danger`); смысл
  несёт подпись под иконкой. Тач-таргет ≥ 44px, `role="toolbar"`,
  `useReducedMotion`.

## Анатомия списка с фильтром

Порядок сверху вниз (эталоны — `components/Ticket/Filter.jsx` и
`components/KnowledgeBase/Explorer.jsx`):

1. Главное действие («Новая заметка») и **поиск** (`UI/SearchBar`).
2. **Сегментный переключатель набора данных** (`ToggleButtonGroup type="radio"`)
   там, где наборы взаимоисключающие («Активные | Архив»). Два независимых
   свитча для взаимоисключающих наборов — ошибка: их приходится синхронизировать
   руками, и пользователь не видит, что они спорят.
3. **Чипы-фасеты со счётчиками** (`ToggleButton type="checkbox"`), счётчик — по
   загруженному набору.
4. **Очереди/спецфильтры** — только тем, у кого есть права.
5. **`Accordion` «Фильтры»** с бейджем числа активных — прячет длинные
   мультиселекты и отдаёт вертикаль списку.
6. **Сортировка** — текст-дропдаун (в `app/ListWrapper` встроена; легаси —
   `UI/Select`). Счётчик — у заголовка страницы (легаси — «Найдено: N»).
7. Список; при группировке — заголовки групп со счётчиком
   (`app/ListGroupLabel`; статусные группы — язык статус-борда, как в
   «Вендорах»: активные бирюзой, отключённые гаснут).

Правила:

- **Не рендерить заголовок секции, если её содержимое скрыто правами.** Пустой
  `<h5>Модерация</h5>` между двумя `<hr>` у обычного сотрудника — это баг.
- **Список по умолчанию показывает всё, что пользователю видно.** Не требовать
  запроса или фильтра, чтобы данные вообще появились: бэкенд уже отдал ровно
  разрешённое, а фильтр — инструмент сужения, а не открывания.
- **Группировка выключается сама, если группа осталась одна** — заголовок с
  единственной группой не несёт информации.
- **Пустое состояние предлагает действие**: «Ничего не нашлось. Измените запрос
  или сбросьте фильтры», а не «Список пуст».
- Мобильная страница списка собирается тем же `ListWrapper` (в мигрированном —
  поиск всегда виден, кнопка «Фильтр» → Sheet с `FilterContainer`, сортировка
  справа). Если страница рендерит `<Outlet/>` сама — передайте
  `renderOutlet={false}`, иначе маршрут смонтируется дважды.
- **Фильтр есть на каждой странице справочника**: Sheet (`filter` проп
  `ListWrapper`) с полным набором, общий для десктопа и мобайла. Дублирующие
  чипы-переключатели в строке инструментов не ставим — применённое видно в
  плашке бейджей (`app/FilterChip` остаётся для мест вне `ListWrapper`).
- **Ключевой фасет сущности** (взаимоисключающие значения) выводим одним из двух
  способов — выбор фиксируем при согласовании макета, единообразно в пределах
  страницы:
  - **дропдаун-чип `app/ChipSelect` в строке инструментов** (`toolbar` проп) —
    быстрый доступ рядом с поиском; выбранное видно в самом чипе, сброс —
    пунктом «Все …». Живой пример: «Тип устройства» на странице атрибутов
    (`pages/DeviceAttribute/List.jsx`);
  - **одиночный `UI/Select` внутри Sheet-фильтра** — когда фасет уместнее рядом
    с прочими условиями, а не отдельным чипом; первая опция «Любое/Все …»
    сбрасывает фасет. Живой пример: «Назначение» на странице типов
    (`components/DeviceType/Filter.jsx`; «Основные устройства» = тип без
    спец-флагов). Список значений фасета выносим в отдельный модуль
    (`components/DeviceType/kinds.js`), раз его читают и фильтр, и бейдж.
  В обоих случаях применённое значение дублируется **снимаемым бейджем** в
  липкой плашке (`activeFilters`), а «Дублирующих чипов-переключателей» правило
  выше не запрещает — оно про повторную индикацию уже применённого, а не про
  сам ключевой фасет.
- **Применённые фильтры всегда видны**: страница передаёт в `ListWrapper`
  `activeFilters=[{ key, label, onRemove }]` — липкий «остров» бейджей над
  списком (виден и при проскроленном в конец списке), каждый бейдж снимается
  крестиком, рядом «Сбросить» (сбрасывает стор и очищает поиск). Лейблы —
  человеческие («Тип: Число»), не значения полей.
- Пустые состояния списков встроены в `ListWrapper`: «Ничего не нашлось» +
  «Сбросить фильтры»/«Открыть фильтр» при отфильтрованном в ноль; «Список
  пуст» + «Добавить …» при пустых данных.

### Бюджет управляющих элементов

Считайте пиксели до первой строки данных. Каждый контрол в этом списке стоит
40–50 px, и все они кажутся полезными по отдельности: кнопка создания, поиск,
переключатель набора, чипы фасетов, очереди, аккордеон фильтров, сортировка.
Семь рядов — и в колонке шириной в треть экрана видно три записи из двухсот.

- **Главное действие — иконка рядом с поиском**, а не залитая кнопка во всю
  ширину. Ряд экономится целиком.
- **Редко трогаемое — под аккордеон «Фильтры»**, а бейдж считает всё скрытое,
  включая набор данных и выключенные фасеты. Применённый фильтр не бывает
  невидимым.
- **Набор одинаковых кнопок-режимов — в одну кнопку-меню со счётчиком.** Четыре
  очереди модерации с заголовком секции — это ~100 px; иконка со значком `5` и
  дропдаун — 31 px, и разбирают очередь не каждый день.
- **Сортировка — ссылка-дропдаун**, а не полноразмерный `Select` на 38 px: её
  меняют раз в сессию.
- **Строка состояния вместо всегда видимых индикаторов**: контекст («Архив»,
  «Очередь: На проверку»), «Найдено: N», «Сбросить» — только когда есть что
  сбрасывать. В чистом состоянии эта строка почти пуста и ничего не стоит.

Эталон — `components/KnowledgeBase/Explorer.jsx`: было ~370 px до первой
заметки, стало ~110 px.

### Когда список перестаёт помещаться на экран

Сотня строк — ещё список. Несколько сотен — уже дерево, и вертикаль кончается
раньше данных. Три правила, в этом порядке:

1. **Группы свёрнуты по умолчанию — все.** Список свёрнутых групп помещается на
   экран целиком и сам работает навигацией. Раскрываются автоматически только
   две вещи: группа открытой сейчас сущности (иначе deep-link ведёт в невидимую
   строку) и — пока идёт поиск — группы с совпадениями.
2. **Раскрытие поиском — настоящее, а не подменённый флаг.** Заголовок группы,
   который выглядит кликабельным и не реагирует на клик, — баг. Раскрывайте
   группы вызовом, а при очистке запроса сворачивайте обратно: их раскрыл поиск,
   а не человек.
3. **На узком экране дерево превращается в drill-down.** Раскрытая группа на
   мобилке не влезает, а строка «Студия · 21» влезает вся: сначала список групп
   со счётчиками, тапом — содержимое группы, тапом — сущность. Поиск и очереди
   на модерацию из drill-down выпрыгивают в плоский список: там ось группировки
   не отвечает на заданный вопрос.

Виртуализация на этом масштабе не нужна и не решает проблему: тормозит не
браузер, а глаз. Начинайте с того, что уменьшает число строк, а не с того, что
ускоряет их отрисовку.

### Поиск переезжает на сервер вместе с данными, по которым ищет

Клиентский поиск бесплатен, только пока искать не по чему. Как только в ответе
списка появляется полный текст сущности (`plainText`, `content`, описание) —
каждый заход в раздел скачивает его целиком, ради `String.includes()` у тех
немногих, кто вообще что-то ищет.

Признак, что пора: **в списке приезжает поле, которое список не показывает.**
Тогда — `?search=` на бэкенде, поле выкидывается проекцией, а на клиенте:

- инпут обновляется мгновенно, запрос — с задержкой (300 мс);
- терм экранируется (`escapeRegex`) и ограничивается по количеству, иначе `.*`
  в поисковой строке становится сканом коллекции;
- **выбранные значения фасетов хранятся объектами, а не id.** Серверный поиск
  сужает набор, из которого собираются опции селекта; по одному id восстановить
  пилюлю выбранного значения будет уже нечем, и выбранная компания исчезнет из
  собственного фильтра, стоит начать печатать.

## Формы → нижняя шторка

Это центральный паттерн приложения. Форма добавления/редактирования — **вложенный
маршрут** (`add`, `update/:id`) списка, который рендерится в **шторке,
выезжающей снизу** (десктоп — колонка 672px по центру, мобайл — почти весь
экран).

**Механика (общая для обеих систем):**

- Состояние видимости — общий стор `store/offcanvas.js`
  (`isActive` / `setShow` / `setClose`); `layout/Root.jsx` сам открывает и
  закрывает шторку по хвосту пути (`add`/`update`/`schedule`/`confirm`).
  **Исключение:** формы базы знаний открываются в основной панели.
- Мигрированные экраны: шторку с `<Outlet/>` рендерит `app/ListWrapper`
  (внутри — `app/FormSheet`); кнопка «Добавить …» встроена в его шапку.
  Легаси: `UI/ListWrapper.jsx` (bootstrap-Offcanvas) + `UI/AddButton.jsx`.
- **Клик по свободной области экрана и `Escape` не закрывают шторку формы на
  десктопе** (`app/FormSheet` гасит `onInteractOutside`/`onEscapeKeyDown` при
  `!isMobile`) — защита от потери данных в наполовину заполненной форме. Закрыть
  можно **крестиком** шторки или **успешным сабмитом** (оба идут через
  `onOpenChange`). На мобайле — поведение по умолчанию (тап по затемнению
  закрывает).
- **Ширина шторки на десктопе — под контент**: по умолчанию колонка 672px
  (`max-w-2xl`); большим формам со сводкой даём `wide` (`max-w-4xl`, 896px) —
  проп идёт `app/ListWrapper` → `FormSheet`. Мобайл — всегда почти весь экран.

**Оболочка формы — `components/app/FormWrapper`** (легаси —
`UI/FormWrapper.jsx` с той же механикой):

- `fetcher.Form method="post"` (router-actions), сабмит без перезагрузки.
- Заголовок формы; кнопка закрытия — крестик самой шторки.
- Ошибки/успех — `AlertMessage` (`variant="danger"`/`"success"`).
- Кнопки справа внизу: **«Отмена»** (ghost) и **«Сохранить»** (primary,
  `disabled` пока идёт сабмит); на мобайле ряд липнет к низу шторки.
- При успешном сабмите шторка закрывается и происходит `navigate("..")`
  (или `successTo`).

**Раскладка полей внутри формы:**

- Каждое поле — **`app/Field`** (лейбл + `required`-звёздочка + контрол +
  `hint`); свитчи — **`app/SwitchField`** (подпись + подсказка-эффект + скрытый
  input `"true"/"false"` для FormData). Руками эти блоки не собираем.
- Выпадающие списки — `UI/Select` + скрытый `<input type="hidden" name=...>`,
  чтобы значение попало в `FormData` router-экшена.
- Два поля в ряд — `tw:grid tw:gap-3 tw:md:grid-cols-2`; узкие экраны
  схлопываются в один столбец.
- Длинную форму делим на секции `ui/Card` с заголовком — это способ «разбить»
  форму без модалок-вкладок (легаси-эталон структуры —
  `components/ClientDevice/Form.jsx`).
- Даты — нативный `<input type="date">`; перед записью в стейт приводим к
  `yyyy-MM-dd`.

### Навигация после сабмита (куда попадает пользователь)

Правило трёх исходов — его нарушение дезориентирует («создал модель с
карточки типа — оказался на списке моделей»):

1. **Создание сущности, у которой есть карточка, → карточка созданной
   сущности**, а не список. `successTo` у `FormWrapper` принимает функцию от
   ответа action — она строит адрес из `_id` созданного объекта (значит,
   бэкенд-ответ обязан содержать созданную сущность; наши `add`-контроллеры
   уже так отвечают). Эталон — `pages/DeviceModel/Add.jsx`,
   `pages/DeviceType/Add.jsx`. У сущности без карточки (вендор, атрибут)
   исход по умолчанию — остаться на списке (`".."`).
2. **Редактирование → остаёмся там, где были.** Форма правки — вложенный
   маршрут той страницы, где её открыли: у списка — `update/:id` в его
   шторке, у карточки — `update` в шторке самой карточки (после сабмита
   `".."` возвращает на неё). Кнопка «Изменить» на карточке никогда не
   уводит на список (`components/DeviceModel/View.jsx`,
   `components/DeviceType/View.jsx`).
3. **Создание с «чужой» страницы → форма открывается вложенным маршрутом
   этой страницы.** «Добавить модель» на карточке типа — это `models/add`
   на карточке типа (шторка на месте, контекст не теряется), а не переход
   на список моделей; после успеха — п.1, карточка созданной модели.

Механика в `app/FormWrapper`: по успеху `navigate(successTo ?? "..",
{ replace: true })` — запись формы не остаётся в истории, поэтому браузерное
«назад» с карточки созданной сущности возвращает на страницу, где нажали
«Добавить». Дополнительно карточка даёт постоянный путь вверх по иерархии:
родительская сущность в hero — ссылка (тип устройства на карточке модели),
а не просто текст.

## Диалоги → только мелкие вещи

Модальные диалоги — для **подтверждений и коротких действий**, не для форм.

Эталон (мигрированные) — **`components/app/DeleteItem.tsx`**:

- `AlertDialog` + `RouterForm method="post"`: заголовок — название удаляемого,
  текст — **«Вы уверены? Это действие нельзя отменить.»**, скрытый `id`,
  футер — «Закрыть» и **«Удалить»** (`destructive`, `intent="delete"`).
- Из «⋯»-меню диалог открывается **состоянием** (`DeleteDialog` рендерится вне
  radix-меню — меню размонтирует содержимое при закрытии), это уже собрано в
  `app/ListRow`.
- Диалог закрывается на сабмите: строка исчезнет с редиректом экшена, и
  открытый модальный radix оставил бы залипший `pointer-events` на `<body>`.

Легаси-эталон — `components/DeleteItem.jsx` (bootstrap-Modal, та же механика);
так же оформлены действия над тикетом (`components/Ticket/Actions/*`). Если
полей становится много или появляется логика в несколько экранов — это уже не
диалог, переносите в шторку/мастер.

**Не делать:** длинные формы создания/редактирования в диалоге; вложенные
диалоги; диалог ради того, что укладывается в шторку снизу.

## Большие многошаговые формы → горизонтальный прогресс

Для мастеров (несколько логических шагов) сверху показываем **горизонтальный
индикатор прогресса**, чтобы пользователь видел, где он и сколько осталось.
**tw-эталон** — `components/ServicePlan/Form.jsx` (мастер услуги, макет
согласован до кода).

**Когда дробить форму на шаги.** Если полей много и они распадаются на
**логические этапы** (у услуги: «Основное» · «Тарификация» · «График») — это
мастер, а не одна простыня. Форму разбиваем на шаги с валидацией по шагу и
кнопками «Назад»/«Далее»/«Сохранить» (эталон механики — `ClientDevice/Form.jsx`:
массив `STEPS`, `stepValid(i)`, `maxReached`, `attempted`). Длинную, но
**одно-логическую** форму мастером не притворяем — её достаточно разбить на
секции (см. конец раздела).

**Степпер — `components/app/WizardStepper.tsx`** (tw, готовый):

- Ряд кружков 32px с номером; пройденные — `RiCheckLine`. Состояния токенами:
  текущий/пройденный — `bg-primary text-white`; будущий — `bg-accent text-faint`
  + `inset-ring`; подписи — `accent-text` / `muted-foreground` / `faint`.
- Между кружками — линия-коннектор (`bg-primary` для пройденного участка,
  `bg-border` для оставшегося).
- Клик по шагу — в режиме правки (`allowJump`) или по достигнутым (`maxReached`).
- **По умолчанию тянется на всю ширину контейнера** (`flex` + дети `flex-1`). На
  форме со сводкой это обязательно: степпер идёт **во всю ширину шторки**, а не
  обрывается над левой колонкой — иначе лента шагов смотрится оторванной от
  сводки. Нужен узкий — задайте `max-w-*` через `className`.

  ```jsx
  <WizardStepper
    steps={[{ label: "Основное" }, { label: "Тарификация" }, { label: "График" }]}
    current={step}
    maxReached={maxReached}
    allowJump={isEdit}
    onStepClick={handleStepClick}
  />
  ```

**Живая сводка справа — у большой формы-мастера обязательна.** Панель
(`components/ServicePlan/Summary.jsx`) наполняется по мере прохождения шагов
(незаданное — «—», разделы раскрываются по `maxReached`) и держит контекст:
название, категории, цена, график видны сразу, не листая шаги назад. Под сводку
шторку **расширяем** (`wide` у `ListWrapper`/`FormSheet` → `max-w-4xl`), колонки —
`tw:flex tw:flex-col tw:md:flex-row` (контент `flex-1`, сводка `md:w-72`).

**Легаси-степпер** — `components/ClientDevice/WizardStepper.jsx` (bootstrap,
проп `currentStep`; состояния `bg-success`/`bg-primary`/`bg-secondary-subtle`) —
остаётся на немигрированных мастерах.

**Прогресс-бар — `ProgressBar` из react-bootstrap** (живой пример —
`components/Ticket/View/AiGuide.jsx`): когда нужен процент выполнения/готовности,
а не дискретные шаги.

- Тонкая полоса (`style={{ height: "0.4rem" }}`), `now={progress}`.
- Над ней — компактная подпись и счётчик «N / M».
- Вариант по состоянию: `progress === 100 ? "success" : ...`.

> ⚠️ В `AiGuide.jsx` для незавершённого прогресса исторически стоит
> `variant="info"`. В тёмной теме это плохо читается — для новых экранов берите
> `primary` (см. следующий раздел).

Длинную, но **одношаговую** форму многошаговой не делаем — достаточно разбить её
на `Card`-секции, как в `ClientDevice/Form.jsx`.

## Панели настроек: переключатель + условные блоки

Экраны настроек с выбором внешнего сервиса (эталон —
`components/Preferences/Ai.jsx`, раздел «Искусственный интеллект») строятся по
повторяющемуся паттерну «селектор → условный блок конфигурации».

- **Мастер-переключатель** `Form.Check type="switch"` включает раздел; пока он
  выключен, все поля ниже идут `disabled`.
- **`Form.Select` выбирает вариант** (провайдера) и раскрывает его блок настроек
  через `{provider === "..." && (<>…</>)}`. У каждого варианта — свой набор
  полей: секреты всегда `Form.Control type="password"` (API-ключи), а
  специфичные поля добавляются точечно (например, «folder ID» у Yandex,
  помеченный `*`).
- **Динамический список «по ключу»** — когда варианты подгружаются от сервиса по
  введённому ключу: `Form.Select` + кнопка обновления рядом (`RiRefreshLine`, на
  время запроса — `Spinner`). Плейсхолдер «— загрузите список —», уже
  сохранённое значение остаётся выбираемым (даже если его нет в свежем списке),
  ошибка загрузки — инлайн `Form.Text className="text-danger"`.
- Если у варианта **нет API для перечня опций**, вместо динамического поля —
  либо обычный `Form.Select` со статическим набором (как у Yandex SpeechKit и
  YandexGPT; бэкенд отдаёт фиксированный список), либо, когда каталог открытый
  или слишком большой, свободный ввод `Form.Control type="text"` с подсказками
  через `<datalist>` (как у Yandex AI Studio — идентификатор модели вводят
  вручную).
- Подсказки под полями — `Form.Text className="text-muted"`.
- Эти панели используют **нативный `Form.Select`**, а не обёртку `UI/Select`:
  это контролируемая конфигурация, которую сохраняют целиком одним объектом, а
  не router-action форма (там, наоборот, обязателен `UI/Select` + скрытый input).

## Страница личных настроек («Мой аккаунт», мигрирована 2026-07)

Эталон паттерна «настройки» в новой системе — `pages/User/MyAccount.jsx`:

- **Одна страница вместо вкладок**: разделы — `app/SettingsSection`-панели
  подряд; на десктопе слева липкий рейл-якорь со scrollspy (слушатель скролла
  на `window` — только в `BrowserView`), на мобайле — просто секции.
- Строки настроек — `app/SettingRow` (название + hint + контрол справа).
- **Уведомления — матрица «событие × канал»** (shadcn Checkbox): 6 строк
  вместо двух списков свитчей. Скрытые глобальными настройками категории всё
  равно уходят в FormData скрытыми полями с текущими значениями — иначе
  сохранение молча сбрасывает их (закрытый баг легаси).
- Каждая секция-форма — свой `fetcher.Form` с `intent` на action страницы;
  результат — глобальный тост (`store/toast-store.showToast`).
- Тема — сегмент трёх состояний (light/dark/system) через `ThemeContext`;
  до эндшпиля миграции смена темы перезагружает страницу (как в навбаре).
- **Фоновая картинка пользователя**: читаемость текстов канвы обеспечивает
  «лист» в `layout/Root.jsx` (см. ниже), сами экраны про фон не знают.
- **Подтверждение внешнего события** (подключение Telegram-бота): после
  действия — revalidate-поллинг loader'а до смены статуса, с гейтом
  `useFetchers().every(f => f.state === "idle")` — ревалидация при живом
  fetcher роняет роутер («Did not find corresponding fetcher result»).
  Статус ожидания — текст с точкой `warning` и подсказкой, что сделать.

Бэкенд-контракты, на которые опирается экран (закреплены после починки):

- `updateMyAccount` **мержит** `notify` по путям, а не заменяет объект —
  частичная форма не сбрасывает чужие категории и не трогает `telegramBot`
  (привязка бота создаётся только ботом через `/api/tg/auth`).
- **Имя категории уведомления едино в трёх местах**: `prefs.notify.personal.X`,
  `user.notify.byTelegram.X`, `user.notify.byEmail.X` — так его читают
  `notifyTg`/`notifyEmail` в `middleware/notifications.js`. Новая категория =
  флаг в модели Preferences (+ свитч в админке) + поля в модели User + ветка
  `case` в цикле нотификаций; значение `lastAction` берётся из **enum схемы
  Ticket**.
- **Статика, читаемая на каждой странице** (фон рабочего стола), хранится
  **локально** (`uploads/`, multer `diskStorage`), не в S3 — иначе каждый вход
  в приложение оплачивается presigned-чтением. Аватары пока в S3 — кандидат
  на ту же схему.

## Тема и цвет

Поддерживаются **светлая и тёмная** темы + «системная» (`store/theme-context.jsx`:
`theme` = `light | dark | system`, флаг `isDark`, сохраняется в `localStorage`).
Провайдер ставит класс **`.dark` на `<html>`** — от него живут все токены новой
системы (и `UI/Select`: он стилизован css-переменными и тему в JS не читает).
Легаси-CSS дополнительно переключают `layout/DarkTheme.jsx` /
`layout/LightTheme.jsx` (смена темы пока перезагружает страницу).

**Палитра новой системы — токены** (`src/styles/tailwind.css`, значения для
светлой / тёмной):

| Токен | Значение | Когда |
|---|---|---|
| `primary` | бирюза `#00bc8c`, текст белый | главное действие («Сохранить», «Добавить …»), одно на экран |
| `background` | `#fff` / `#1d2024` | **базовая поверхность** (шторки, диалоги, панели) — сток-семантика shadcn, НЕ канва |
| канва | `#f4f5f6` / `#16181c` | фон страницы; красится оболочкой (`#root`/`body`), не токеном |
| `card` / `popover` | как `background` | панель списка, меню, поповеры |
| `muted-foreground` / `faint` | два уровня приглушённого | мета-строки, подсказки / счётчики, плейсхолдеры, неактивное |
| `accent` | серый ховер | фон наведения строк и ghost-кнопок |
| `accent-text` | читаемая бирюза `#007a5e` / `#0bd39f` | акцентный текст: метки групп, включённые способности |
| `destructive` | `#d63c3c` / `#f05252` | удаление, ошибки, `required`-звёздочка |
| `success` / `warning` / `info` | `#00bc8c` / `#eb9c14` / `#4d94e8` | семантика статусов (варианты Badge/Alert/Button) |
| `border` / `border-soft` | тонкие границы | контуры панелей / разделители строк |

Принципы: **статус — цветной текст с точкой, а не заливной бейдж**; заливка —
только у главного действия и семантических алертов; цветом на экране говорим
максимум о трёх вещах.

**Палитра легаси-экранов (Bootstrap-варианты):**

| Вариант | Когда |
|---|---|
| `primary` (тёмно-синий `#2c3e50`) | основное действие, активный шаг |
| `success` | успех, положительный статус; акцент мобильной навигации |
| `danger` | удаление, ошибки, метка обязательного поля (`*`) |
| `warning` | предупреждения, «нужно внимание» |
| `secondary` | вторичные действия («Закрыть», «Отмена») |
| `light` | нейтральные плашки: счётчики, пустые состояния |
| `muted` | подсказки, второстепенный текст |

**Правило тёмной темы (важно):** **не использовать текст и иконки цвета `info`**
(`text-info`, иконки с `className="text-info"`) — на тёмном фоне контраст низкий,
читается плохо. Вместо этого:

- акцентный текст/иконки — `primary`, `success` либо нейтральный «корпусный»
  цвет (`text-body` / `text-body-emphasis`), которые сами адаптируются к теме;
- если нужен бейдж `bg="info"` — обязательно задавайте читаемый текст
  (`<Badge bg="info" text="dark">`), не полагайтесь на цвет «по умолчанию».

В коде ещё встречаются `text-info` и `bg="info"` (`AiSpeechBadge`,
`AiCategoryBadge`, виджеты дашборда) — это легаси; в новых экранах не повторяем,
а при касании заменяем. Уже вычищено: бейджи категорий базы знаний (уехали в
`components/KnowledgeBase/BindingChips.jsx`), `ModerationCard`, кнопка выдачи
пользователю на странице устройства. Залитая кнопка `variant="info"` (мобильный
«Фильтр» в `ListWrapper`) контрастна и допустима — правило касается **текста и
иконок**.

**Цвет — ресурс, а не украшение.** Правило, выведенное на редизайне базы знаний:
на экране цветом говорят максимум о трёх вещах. В базе знаний это тип заметки,
состояние проверки и опасное действие. Однородные сущности (категория, компания,
пользователь) различаются **иконкой** и живут одинаковой нейтральной пилюлей
(`Badge bg="secondary"` + `Ri*`), иначе бейджи превращаются в светофор без
смысла.

## Иконки

- Только **react-icons / Remix (`Ri*`)**, размер задаём `size={...}`.
- Закрепившиеся соответствия: `RiSaveLine` — сохранить, `RiArrowGoBackFill` —
  закрыть/назад, `RiAddFill` — добавить, `RiDeleteBinLine` — удалить,
  `RiRefreshLine` — обновить/перегенерировать, `RiCheckLine` — выполнено.
- Иконка обычно идёт перед текстом в кнопке (`<RiSaveLine /> Сохранить`).
- На узких экранах подпись прячут (`d-none d-sm-inline`), оставляя иконку.
- Помним правило тёмной темы: не красить иконки в `info`.

## Обратная связь и статусы

- **Тосты** — **sonner** (`ui/sonner`, `<Toaster/>` в `Root.jsx`, снизу
  справа): вызов — `showToast(variant, message)` из `store/toast-store.js`
  (bootstrap-имена вариантов маппятся на toast.success/error/…). Легаси
  `UI/AlertToast` остаётся только у экранов, рендерящих его локально через
  пропсы, — уходит с миграцией их областей.
- **Инлайн-алерты** — `app/AlertMessage` (мигрированные; принимает
  bootstrap-имена вариантов) / `UI/AlertMessage*` (легаси). Внутри форм
  ошибку/успех показывает сам `FormWrapper`.
- **Статусы в списках** — цветной текст с точкой или метка группы
  (`ListGroupLabel`); заливной `Badge` — для компактных мест, карты
  «статус → вариант» собираем рядом с компонентом.
- **Длительные операции** — `app/Spinner` / легаси `animations/Spinner`; для
  опроса фонового результата паттерн «спиннер + polling» (см. `AiGuide.jsx`).
- **Пустые состояния** — `AlertMessage variant="light"` с понятным текстом
  («Список пуст»), не пустой экран.

## Чек-лист перед мерджем UI

- [ ] Форма создания/редактирования открывается в **нижней шторке** через
      маршрут `add`/`update` и обёрнута в `FormWrapper` (мигрированные —
      `components/app`, легаси — `UI/`). **Исключение — документ**: он правится
      на месте (см. «Правка на месте»).
- [ ] Шторка формы на десктопе **не закрывается кликом по свободной области и
      `Escape`** (закрытие — крестик или успешный сабмит); ширина под контент
      (`wide` для больших форм со сводкой). На мобайле — поведение по умолчанию.
- [ ] Навигация после сабмита: **создание → карточка созданной сущности**
      (successTo-функция из ответа action), **правка → остаёмся где были**
      (вложенный `update`-маршрут страницы, где форму открыли); redirect — с
      `replace`, «назад» возвращает туда, где нажали «Добавить». Форма с
      «чужой» страницы (модель с карточки типа) — вложенным маршрутом этой
      страницы, а не переходом на «родной» список сущности.
- [ ] Диалог используется **только** для подтверждения/короткого действия.
- [ ] Многошаговый сценарий имеет **горизонтальный степпер/прогресс** сверху;
      большая форма-мастер разбита на логические шаги, степпер — во всю ширину
      шторки, справа — **живая сводка**, наполняемая по мере прохождения.
- [ ] Поля — через `app/Field`/`app/SwitchField` (легаси: `*` —
      `text-danger`, подсказки — `text-muted`); выпадающие списки — `UI/Select`
      (+ скрытый input для router-формы).
- [ ] Мигрированный экран: ни одного импорта react-bootstrap и
      bootstrap-классов; утилиты `tw:*` только из сетки (без `-[Npx]`);
      повторяемые куски выделены в `components/app`, а не скопированы.
- [ ] Иконки — из набора `Ri*`; нет `text-info`/иконок цвета `info`
      (проверено в **тёмной** теме).
- [ ] Семантика — вариантами/токенами, без кастомных цветов; цветом говорим
      максимум о трёх вещах; статус в списке — текст с точкой/меткой группы,
      а не светофор бейджей.
- [ ] На странице сущности **одна залитая кнопка**, выбранная состоянием;
      остальное — ghost/outline и icon-only `⋯`. Подписи «Действия» нет.
- [ ] Запросы, ожидающие решения, показаны инлайн-`Alert` с автором и датой, а
      не пунктами меню.
- [ ] Статус с автором/датой/сроком выведен фразой, а не голым бейджем; бэкенд
      отдаёт имена (`populate`), а не `ObjectId`.
- [ ] Правка на месте: геометрия чтения и правки совпадает, есть
      `Ctrl+E`/`Ctrl+S`/`Esc`, `useBlocker` + `beforeunload` на несохранённое,
      побочный эффект сохранения назван до сохранения.
- [ ] Список: заголовок скрытой по правам секции не рендерится; данные видны без
      обязательного фильтра; пустое состояние предлагает действие.
- [ ] Мобильная навигация: панели — флекс/absolute-дети `.mobile-shell` (не
      `position:fixed`), скролл только у `.mobile-shell__scroll`, нет слушателей
      скролла на `window`; на экране один плавающий объект.
- [ ] Заблокированное массовое действие объясняет причину (тултип на десктопе,
      тап-подсказка на мобилке) и называет объекты по имени, не по id.
- [ ] Проверено в светлой и тёмной теме, на десктопе и мобайле — для
      мигрированных экранов playwright-проходкой (скриншоты, клики) до показа.
